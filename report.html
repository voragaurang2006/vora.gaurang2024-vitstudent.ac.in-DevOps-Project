3. Jenkins Automation & CI/CD Declarative Pipeline

An automated build pipeline triggers on every git commit. The Jenkins server runs a declarative pipeline that compiles code, executes tests, packages the jar, builds a container image, pushes it to Docker Hub, and rolls out the deployment to the Kubernetes cluster.

Declarative Pipeline Script (Jenkinsfile)

pipeline {
    agent any
    environment {
        DOCKER_IMAGE = "voragaurang2006/corporate-website"
        DOCKER_TAG   = "${BUILD_NUMBER}"
    }
    stages {
        stage('Checkout') {
            steps { checkout scm }
        }
        stage('Maven Compile & Test') {
            steps { bat 'mvn clean test' }
        }
        stage('Maven Package') {
            steps { bat 'mvn package -DskipTests' }
        }
        stage('Docker Build') {
            steps {
                bat "docker build -t ${DOCKER_IMAGE}:${DOCKER_TAG} ."
                bat "docker tag ${DOCKER_IMAGE}:${DOCKER_TAG} ${DOCKER_IMAGE}:latest"
            }
        }
        stage('Docker Hub Publish') {
            steps {
                withCredentials([usernamePassword(credentialsId: 'docker-hub', usernameVariable: 'U', passwordVariable: 'P')]) {
                    bat "docker login -u %U% -p %P%"
                    bat "docker push ${DOCKER_IMAGE}:${DOCKER_TAG}"
                }
            }
        }
        stage('Kubernetes Deploy') {
            steps {
                bat "kubectl apply -f k8s/deployment.yaml"
                bat "kubectl apply -f k8s/service.yaml"
            }
        }
    }
}

4. Application Containerization with Docker

To achieve consistency across deployment environments, we containerized the website using a multi-stage Docker build process. The first stage uses a Maven image to compile and package the source code. The second stage copies the resulting JAR file and executes it within a lightweight OpenJDK JRE runtime environment, ensuring minimal image size and maximum security.

Dockerfile Specifications

# Stage 1: Build JAR package
FROM maven:3.8.4-openjdk-8-slim AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Create lightweight execution container
FROM openjdk:8-jre-slim
WORKDIR /app
COPY --from=build /app/target/corporate-website-1.0.0.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

5. Kubernetes Orchestration & NodePort Exposure

The containerized application is deployed to a Kubernetes cluster. A rolling update strategy ensures zero-downtime deployments. A NodePort service exposes the application to external traffic on host port 30080.

Deployment Manifest (deployment.yaml)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: corporate-website-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: corporate-website
  template:
    metadata:
      labels:
        app: corporate-website
    spec:
      containers:
      - name: corporate-website
        image: voragaurang2006/corporate-website:latest
        ports:
        - containerPort: 8080
        livenessProbe:
          httpGet:
            path: /api/health
            port: 8080
          initialDelaySeconds: 30
        readinessProbe:
          httpGet:
            path: /api/health
            port: 8080
          initialDelaySeconds: 20
